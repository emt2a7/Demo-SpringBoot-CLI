# utility-spec.prompt.md

# 架構師解析：這套架構強在哪裡？
- 第 1 章的「職責外 (Out-of-Scope)」是神來之筆：
- 在企業實務中，Utility 類別最容易變成「垃圾桶（God Class）」。
- 什麼邏輯都往裡面塞，最後 StringUtil 裡面竟然出現了寄送 Email 的程式碼。加上「職責外」的定義，能從源頭掐斷這種壞味道。

- 第 4 章的「邊界條件」直接驅動單元測試：
- 把 Edge Cases 獨立成一個章節，AI 在閱讀這份 spec.md 產生 tasks.md 或單元測試時，會直接把這些 Null、0、極端值轉換成測試案例。
- 這保證了你的 Utility 具備防彈等級的強健度。

- 第 5 章的「零外部依賴」：
- 確保了這個 Utility 是最純粹的 Java Code。
- 這意味著它的測試執行速度會極快（不需啟動 Spring Boot），而且未來如果你想把這個 Util 抽成一個獨立的 .jar 包給其他專案共用，將會非常輕鬆。

# 下一步建議：
- 你可以建立一個名為 .github/prompts/speckit.specify-util.prompt.md 的專屬 Agent。
- 以後只要你需要寫各種 Util（例如 FileUtil, CryptoUtil, MoneyUtil），直接呼叫 /speckit.specify-util，AI 就會套用這套「工具類別專屬架構」來幫你產出無懈可擊的規格書了！
